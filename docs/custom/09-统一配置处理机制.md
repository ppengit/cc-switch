# 统一配置处理机制（最终版）

## 1. 目标

统一目标不是把所有应用写成同一种格式，而是让所有配置动作遵守同一条可解释链路：

- 供应商真实配置只保存在数据库供应商记录中。
- MCP 只由 MCP 管理数据决定。
- 应用 live 文件由当前运行模式生成。
- 代理接管期间 live 文件只连接本地代理，不暴露真实供应商。
- 故障转移由代理运行时选择供应商，不依赖反复改写本地 CLI 配置。

## 2. 三个事实源

### 2.1 供应商配置

供应商配置是 API 请求地址、API Key、模型、协议、额外参数的唯一事实源。

添加/编辑供应商时保存的内容会直接进入 `providers.settings_config`。普通切换、停止接管恢复、代理转发请求都以它为依据。

### 2.2 MCP 配置

MCP 配置由 MCP 管理页维护，不再从供应商配置中隐式推断共享片段。

写 live 时：

- Claude：按 Claude 独立 MCP 机制同步。
- Codex：重建 `[mcp_servers]`。
- Gemini：重建 `mcpServers`。
- OpenCode / OpenClaw / Hermes：按应用自身结构维护对应 provider/MCP 配置。

### 2.3 应用接入配置模板

应用接入配置模板只在代理接管模式下使用，用来把目标应用接入 cc-switch 本地代理。

它不参与普通直连写入，也不决定供应商真实配置。

## 3. 普通直连链路

普通直连模式下：

1. 选择当前供应商。
2. 读取该供应商真实配置。
3. 合并当前应用启用的 MCP 集合。
4. 写入目标应用 live 文件。

如果用户在外部终端启动应用，应用会直接读取这些真实供应商配置。

## 4. 代理接管链路

代理接管模式下：

1. 选择当前供应商。
2. 读取该供应商真实配置并合并 MCP。
3. 更新 live backup。
4. 使用应用接入配置模板生成 live 文件。
5. live 文件只包含本地代理地址和占位 Token。
6. 真实请求由 cc-switch 代理读取当前目标供应商后转发。

接管 live 的安全要求：

- Claude：`ANTHROPIC_BASE_URL` 指向本地代理，Token 为 `PROXY_MANAGED`，模型默认由接入模板控制。
- Codex：`auth.json` 使用 `PROXY_MANAGED`，`config.toml` 中 active provider 的 `base_url` 指向本地代理 `/v1`。
- Gemini：`.env` 中 `GOOGLE_GEMINI_BASE_URL` 指向本地代理，`GEMINI_API_KEY=PROXY_MANAGED`。

## 5. 故障转移链路

故障转移只承诺在代理接管模式下完整可靠。

原因：用户可能从任意外部终端启动 Claude/Codex/Gemini，如果 live 文件不是本地代理地址，cc-switch 无法拦截请求，也无法在熔断后切换供应商。

故障转移发生时：

- 本地应用 live 文件保持稳定，不反复写入 A 或 B 的真实供应商地址。
- 代理路由器选择新的上游供应商。
- 上游请求使用新供应商的真实配置。
- UI 的当前代理、处理中、熔断、降级状态都从代理运行时状态读取。

## 6. 模型字段的意义

供应商模型字段保留且有意义，但它不是所有场景下的强制最终模型。

- 直连模式：模型字段通常会写入应用 live，作为 CLI 默认模型。
- 代理接管：请求体自带 `model` 时，请求体优先；供应商模型字段用于默认值、模型映射和展示。
- Claude：默认模型映射可把 Haiku/Sonnet/Opus 请求映射到供应商支持的实际模型。
- 用量统计：保留请求模型、响应模型和映射后的模型，用于价格匹配与日志解释。

## 7. 参数归属原则

供应商参数应只放“上游请求相关”的内容。

如果某个参数会影响本地 CLI 启动、沙箱、审批策略、终端行为或应用自身行为，它应属于应用接入配置模板或应用级 live 配置，而不是供应商私有配置。

这样可以避免故障转移时出现“代理已经切到 B，但本地 CLI 仍按 A 的启动参数运行”的错觉。

## 8. 稳定性约束

- 模板保存时如果应用正在接管，只重建接管 live，不写入真实供应商地址。
- MCP 变更时同时更新 live backup 和接管 live。
- Codex 代理接管会清理真实 `base_url`，只留下本地代理地址。
- Gemini 无启用 MCP 时不写空 `mcpServers`。
- 请求活动状态只保存在内存，不进入数据库，用于快速判断应用是否正在工作。

## 9. 实时模型来源规则

实时活动条、供应商列表“处理中”状态中的模型字段遵守以下刷新顺序：

1. 请求刚进入代理时，先记录客户端请求上下文中的模型。
2. 请求真正发往上游前，用过滤和映射后的实际出站请求体 `model` 覆盖一次。
3. 如果流式事件暴露了更准确的响应模型，再按流式事件继续刷新。
4. 如果最终非流式响应体带有模型字段，再以最终响应为准。

这样做的目的：

- 避免把供应商默认模型误当成实际处理模型。
- 避免 Codex / Claude 转换流里一直显示初始请求模型。
- 在上游把请求重写为实际能力模型时，UI 能尽快反映真实处理方。
