# 统一配置处理机制

## 1. 目标

将六个应用的供应商配置处理统一到同一条逻辑链路：

- 供应商原始配置
- 可选配置模板
- 有效配置生成
- live 文件写入
- 代理接管 / 恢复 / 热切换

统一的是处理机制，不是强行把所有应用写成同一种文件格式。不同应用仍保留各自官方配置格式，但 CC Switch 内部不再混用“有的做 merge，有的直接覆盖，有的完全跳过”这类不一致行为。

## 2. 当前统一后的规则

### 2.1 配置模板

以下应用现在都使用同一套“供应商原始配置 -> 可选模板渲染 -> live 写入适配器”的机制：

- Claude
- Codex
- Gemini
- OpenCode
- OpenClaw
- Hermes

说明：

- `meta.useConfigTemplate` 是当前主开关，默认 `true`。
- `true`：根据应用模板替换 `{providerConfig}` / `{mcpConfig}` / `{settingsConfig}`。
- `false`：跳过模板，直接写供应商原始配置。
- 默认模板保持最小侵入，通常等价于 `{providerConfig}`。
- Codex 的 `{mcpConfig}` 自带 `[mcp_servers]` 表头，仅当存在启用的 MCP 时输出。
- Gemini 的 `.env` 与 `settings.json` 是两个独立模板文件；仅当存在启用的 MCP 时输出 `mcpServers`，否则不写空 `mcpServers`。

### 2.2 旧通用配置兼容

旧“通用配置”功能保留底层兼容函数，但不再作为新增配置的主流程：

- 新增/编辑供应商表单不再展示旧通用配置编辑入口。
- 保存供应商时不再剥离公共片段，数据库中的供应商配置保持用户输入的原始事实源。
- 读取 live 配置不会反向覆盖正在编辑的供应商，避免把模板渲染结果写回数据库。
- 旧字段 `commonConfigEnabled` 仅用于历史数据兼容，不参与新模板决策。

### 2.3 Gemini 的文本保真

Gemini `.env` 现在支持原始文本保真写入：

- 模板渲染后的注释、分段、顺序会跟随有效配置一起保留
- 正常写 live 时，如果原始文本和 `env` 键值对一致，则直接写原始文本
- 如果代理接管、热切换等流程修改了 `env` 导致文本失效，则自动回退到安全的序列化写法
- 从 live 读取、备份、恢复时会尽量携带原始 `.env` 文本，减少注释丢失

### 2.4 代理接管期间的安全写入

代理接管开启时，live 文件不能直接写回真实供应商地址和真实 Key，否则会绕开代理并破坏故障转移。当前统一规则为：

- 供应商变更、配置模板保存、热切换都会先根据数据库供应商生成有效配置。
- 再更新 live 备份，保证停止接管时可恢复到最新供应商配置。
- 最后按应用写入接管态 live：
  - Claude：保留 `ANTHROPIC_BASE_URL` 指向本地代理，Key 使用 `PROXY_MANAGED`。
  - Codex：保留 `config.toml` 中 active model provider 的 `base_url` 指向本地代理 `/v1`，`auth.json` 使用 `PROXY_MANAGED`。
  - Gemini：保留 `GOOGLE_GEMINI_BASE_URL` 指向本地代理，`GEMINI_API_KEY` 使用 `PROXY_MANAGED`。
- 因此模板保存、故障转移、接管代理开启/关闭都走同一套“有效配置 + 接管字段覆盖”的机制。
- Codex 接管态会先清理 `config.toml` 中所有 `base_url`（包括 `model_providers.*.base_url` 和顶层 `base_url`），再只写入本地代理地址，避免真实供应商地址残留在 live 配置。

### 2.5 MCP 变更与代理备份

MCP 管理同样接入统一配置链路，避免“live 已更新、停止接管后又恢复旧 MCP”的问题：

- 新增、编辑、启用、禁用、删除 MCP 后，先按服务器 ID 清理需要移除的 live 项。
- 再按受影响应用重建当前有效配置。
- 对普通 live 模式：重建实际配置并同步当前启用的 MCP 集合。
- 对代理接管模式：更新 live backup 中的真实供应商配置和 MCP 集合，再写入接管态 live。
- Claude 的 MCP 在独立 `.claude.json`，因此接管态也会额外执行 MCP-only 同步。
- Codex / Gemini 的 MCP 在主配置文件内，接管态备份与 live 会同时刷新，但 live 中仍只保留本地代理地址和 `PROXY_MANAGED`。

## 3. 供应商模型字段的实际意义

供应商里配置的模型不是完全无意义，但它的作用分场景：

- 普通 live 模式：模型字段会写入目标应用配置，作为该应用的默认模型。例如 Claude 的 `ANTHROPIC_MODEL`、Codex 的 `model`、Gemini 的 `GEMINI_MODEL`。
- 代理模式：请求体自带 `model` 时，代理不会盲目丢弃请求模型。Claude 路径会根据供应商里的 `ANTHROPIC_MODEL`、`ANTHROPIC_DEFAULT_HAIKU_MODEL`、`ANTHROPIC_DEFAULT_SONNET_MODEL`、`ANTHROPIC_DEFAULT_OPUS_MODEL` 做映射；未配置映射时保持请求原始模型。
- 用量统计：数据库同时保留 `request_model` 与实际/响应模型，计费规则可选择按请求模型或响应模型匹配。
- 结论：模型字段应保留，但 UI 需要避免误导。它不是“所有请求强制使用此模型”，而是“应用默认模型 / 代理映射规则 / 用量统计参考”。

## 4. Hermes 的补齐

Hermes 之前没有真正纳入通用配置提取链路。本轮补齐后：

- 支持从 Hermes 当前供应商提取通用配置片段
- 自动排除明显的供应商私有字段：
  - `name`
  - `model`
  - `base_url`
  - `api_key`
  - `_cc_source`

## 5. 仍然保留的应用差异

统一机制不代表抹平官方格式差异，下面这些差异仍然存在且应保留：

- Claude / Codex / Gemini：当前供应商主导 live
- OpenCode / OpenClaw / Hermes：累加模式，live 中可同时存在多个供应商
- Gemini：天然是多文件配置（`.env` + `settings.json`），无启用 MCP 时不写空 `mcpServers`
- Claude：MCP 仍然是独立文件，不与 `settings.json` 混写
- Hermes：live 为 YAML，且 provider 写入目标是 `custom_providers`

## 6. 下一阶段建议

如果继续往“完全一致体验”推进，建议下一步做的是：

1. 给 Claude 设计 `settings.json` / `.claude.json` 的双文件模板入口
2. 给 OpenCode / OpenClaw / Hermes 增加“整文件模板预览”，区分单供应商片段和完整 live 文件
3. 在“当前配置”页面增加“由模板生成的有效配置预览”，让用户区分：
   - 原始供应商配置
   - 模板替换结果
   - 当前 live 实际配置
