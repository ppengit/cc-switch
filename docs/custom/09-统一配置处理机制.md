# 统一配置处理机制

## 1. 目标

将六个应用的供应商配置处理统一到同一条逻辑链路：

- 供应商原始配置
- 通用配置片段
- 有效配置生成
- live 文件写入
- 代理接管 / 恢复 / 热切换

统一的是处理机制，不是强行把所有应用写成同一种文件格式。不同应用仍保留各自官方配置格式，但 CC Switch 内部不再混用“有的做 merge，有的直接覆盖，有的完全跳过”这类不一致行为。

## 2. 当前统一后的规则

### 2.1 通用配置

以下应用现在都使用同一套“检测是否使用通用配置 / 合并通用配置 / 从供应商配置中剥离通用配置”的机制：

- Claude
- Codex
- Gemini
- OpenCode
- OpenClaw
- Hermes

说明：

- Claude / Gemini / OpenCode / OpenClaw / Hermes 使用 JSON 片段机制
- Codex 使用 TOML 片段机制
- 旧数据仍兼容“子集检测”逻辑
- 新数据优先以 `common_config_enabled` 显式标记为准

### 2.2 迁移逻辑

历史上只有部分应用参与“通用配置迁移”。本轮后：

- 累加模式应用也会参与旧通用配置的迁移判断
- 命中旧片段的供应商会被自动标记 `common_config_enabled=true`
- 同时会把公共部分从供应商自身配置中剥离，避免重复

### 2.3 Gemini 的文本保真

Gemini `.env` 现在支持原始文本保真写入：

- 模板渲染后的注释、分段、顺序会跟随有效配置一起保留
- 正常写 live 时，如果原始文本和 `env` 键值对一致，则直接写原始文本
- 如果代理接管、热切换等流程修改了 `env` 导致文本失效，则自动回退到安全的序列化写法
- 从 live 读取、备份、恢复时会尽量携带原始 `.env` 文本，减少注释丢失

## 3. Hermes 的补齐

Hermes 之前没有真正纳入通用配置提取链路。本轮补齐后：

- 支持从 Hermes 当前供应商提取通用配置片段
- 自动排除明显的供应商私有字段：
  - `name`
  - `model`
  - `base_url`
  - `api_key`
  - `_cc_source`

## 4. 仍然保留的应用差异

统一机制不代表抹平官方格式差异，下面这些差异仍然存在且应保留：

- Claude / Codex / Gemini：当前供应商主导 live
- OpenCode / OpenClaw / Hermes：累加模式，live 中可同时存在多个供应商
- Gemini：天然是多文件配置（`.env` + `settings.json`）
- Claude：MCP 仍然是独立文件，不与 `settings.json` 混写
- Hermes：live 为 YAML，且 provider 写入目标是 `custom_providers`

## 5. 下一阶段建议

如果继续往“完全一致体验”推进，建议下一步做的是：

1. 把多文件模板机制从 Gemini 扩展为通用 section 模式
2. 给 Claude 设计 `settings.json` / `.claude.json` 的双区段模板
3. 给 OpenCode / OpenClaw / Hermes 补齐“模板实际渲染”而不是仅有模板编辑入口
4. 在“当前配置”页面增加“由模板生成的有效配置预览”，让用户区分：
   - 原始供应商配置
   - 通用配置片段
   - 当前 live 实际配置
