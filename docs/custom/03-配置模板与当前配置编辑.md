# 配置模板与当前配置编辑方案（已实施）

## 1. 配置模板

将“通用配置”扩展为“应用配置模板”，模板支持占位符：

- `{providerConfig}`
- `{mcpConfig}`

按应用分别存储（settings 表键：`config_template_{app}`），避免不同应用配置格式混用。

## 2. 模板写入策略（当前实现）

- 不做整段暴力覆盖。
- 采用锚点/注释区间方式进行替换，保证原有文件结构尽量稳定。
- 当某应用未启用 MCP 时，`{mcpConfig}` 输出为空片段（或空对象，取决于目标格式）。
- 当应用未启用代理接管时，实际配置由“模板 + 替换内容”生成。

### 2.1 Codex 实际落地策略（重点）

- 文件：`src-tauri/src/services/provider/live.rs`
- 对 Codex 通用配置改为“锚点区间替换”而非 TOML 结构化 merge。
- 新增 Codex 模板渲染：
  1. 先构建供应商有效配置（含通用配置开关处理）。
  2. 按模板替换 `{providerConfig}` 与 `{mcpConfig}`。
  3. 对渲染结果执行 TOML 有效性校验；失败则回退到供应商原配置（并告警日志）。
  4. 此结果同时用于：
     - 普通模式写 live
     - 代理接管开启后的热切换备份更新
     - 故障转移场景的当前目标配置生成
- 锚点：
  - `# >>> CC-SWITCH COMMON CONFIG START`
  - `# <<< CC-SWITCH COMMON CONFIG END`
- 处理流程：
  1. 先移除旧锚点区间（若存在）。
  2. 再做一次“兼容旧版结构化写法”的清理，避免重复键。
  3. 最后把通用配置片段写入锚点区间。
- 收益：避免 `toml_edit` merge 导致的注释丢失、段落重排、顺序混乱。

### 2.2 兼容旧数据

- 对历史供应商（未显式标记 `common_config_enabled`）仍保留子集检测逻辑。
- 对未使用锚点的旧配置，仍支持按结构化方式移除历史通用片段后再归一化到锚点模式。

### 2.3 Gemini 实际落地策略（已补齐）

- Gemini 模板现已接入实际渲染。
- 默认模板拆分为两个逻辑区段：
  - `# .env`
  - `# settings.json`
- 当前支持占位符：
  - `{providerConfig}`：Gemini `.env` 变量
  - `{settingsConfig}`：`settings.json` 中除 `mcpServers` 外的供应商配置字段
  - `{mcpConfig}`：按 Gemini CLI 实际格式渲染后的 `mcpServers`
- 写入流程：
  1. 先应用“写入通用配置”的 `.env` 片段。
  2. 再按模板渲染 `.env` 与 `settings.json`。
  3. `settings.json` 渲染后会再次合并供应商自身配置，并统一覆盖 `mcpServers`，避免残留旧值。
  4. 渲染结果同时用于：
     - 普通模式写 live
     - 代理接管期间的 Gemini 备份更新
     - 恢复时的完整 live 回写

### 2.4 其他应用当前状态

- Claude/OpenCode/OpenClaw/Hermes 仍保留各自稳定的 app-native live 写入方式。
- 但它们现在已经接入统一的“通用配置片段检测 / 合并 / 剥离 / 迁移”机制，不再是完全割裂的旧逻辑。
- 模板方面：
  - Codex / Gemini 已进入“模板实际渲染 -> 有效配置 -> live 写入”链路
  - OpenCode / OpenClaw / Hermes / Claude 目前已支持模板前端编辑与保存，后续将继续按官方文档分阶段接入实际渲染，确保兼容性与可回滚性

### 2.5 统一机制补充（新增）

- 新增文档：`09-统一配置处理机制.md`
- 重点说明：
  - 六个应用现在共享同一套底层配置处理思路
  - 不同的是最终 live 文件适配器，而不是是否参与配置治理
  - Gemini `.env` 已支持原始文本保真写入，尽量保留注释、分段和顺序

## 3. 当前配置编辑

新增能力：

- 按应用列出一个或多个配置文件（如 Gemini 的 `.env` 与 `config.json`）。
- 读取文件内容到前端编辑。
- 保存回原文件（含基础路径安全校验）。

## 4. 启动回显

应用启动时尝试从实际配置解析 MCP 配置并回显至 MCP 管理。

- 支持标准多 `mcpServers` 结构解析。
- 再按应用格式写入对应 `{mcpConfig}`。

## 5. 前后端命令接线

新增 Tauri 命令并在 `invoke_handler` 注册：

- `list_app_config_files`
- `read_app_config_file`
- `write_app_config_file`
- `get_app_config_template`
- `set_app_config_template`

前端通过 `configApi` 对应方法调用，支持“当前配置”弹窗完成文件枚举、读取、编辑、保存闭环。

## 6. MCP 回显增强（与模板关联）

- 启动时 MCP 导入从“仅空表触发”改为“每次启动增量导入”：
  - 文件：`src-tauri/src/lib.rs`
  - 作用：管理页始终能回显外部实际配置变更。
- MCP JSON 解析支持标准 `mcpServers` 包裹结构，并支持多服务器批量解释：
  - 文件：`src/utils/formatters.ts`
  - 表单处理：`src/components/mcp/McpFormModal.tsx`
  - 校验：`src/components/mcp/useMcpValidation.ts`
