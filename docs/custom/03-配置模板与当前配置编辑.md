# 配置模板与当前配置编辑方案（已实施）

## 1. 配置模板

原“通用配置”已退出主流程，新的主流程是“应用配置模板”。模板按应用、按文件保存，支持占位符：

- `{providerConfig}`
- `{mcpConfig}`
- `{settingsConfig}`（主要用于 Gemini `settings.json`）

按应用分别存储（settings 表键：`config_template_{app}`），值为 `AppConfigTemplateFile[]` 的 JSON 字符串，避免不同应用配置格式混用。

每个模板文件包含：

- `key`：稳定文件标识，例如 `env` / `settings` / `config`
- `label`：界面显示名，例如 `.env` / `settings.json` / `config.toml`
- `content`：模板文本

## 2. 模板写入策略（当前实现）

- 不再使用旧“通用配置”整段融合逻辑。
- 以供应商数据库配置为原始配置，按 `meta.useConfigTemplate` 决定是否套用模板。
- `meta.useConfigTemplate !== false`：实际环境配置由模板替换占位符生成。
- `meta.useConfigTemplate === false`：实际环境配置直接使用供应商原始配置。
- 当某应用未启用 MCP 时，`{mcpConfig}` 输出为空片段；Gemini 不再写空 `mcpServers`。
- 模板保存时可勾选“保存后更新实际配置”，会立即同步当前应用实际配置文件。
- 若当前应用处于代理接管状态，同步不会直接写入真实供应商地址和真实 Key；系统会先更新 live 备份，再把实际 live 文件保持为本地代理地址和 `PROXY_MANAGED` 占位 Key。

### 2.1 Codex 实际落地策略（重点）

- 文件：`src-tauri/src/services/provider/live.rs`
- 新增 Codex 模板渲染：
  1. 先构建供应商有效配置。
  2. 按模板替换 `{providerConfig}` 与 `{mcpConfig}`。
  3. 对渲染结果执行 TOML 有效性校验；失败则回退到供应商原配置（并告警日志）。
  4. 此结果同时用于：
     - 普通模式写 live
     - 代理接管开启后的热切换备份更新
     - 故障转移场景的当前目标配置生成
- `{mcpConfig}` 仅在存在启用的 Codex MCP 时生成完整 `[mcp_servers]` 表；否则为空。
- 接管态写 live 时会清理所有真实 `base_url`，只保留本地代理地址。
- 收益：避免旧 merge 逻辑导致的注释丢失、段落重排、顺序混乱。

### 2.2 兼容旧数据

- 旧“通用配置”字段与辅助函数只作为历史兼容保留，不再参与新增配置主流程。
- 历史 `commonConfigEnabled` 不再决定 live 写入；新逻辑只看 `meta.useConfigTemplate`。
- 供应商数据库中的配置保持用户输入的原始事实源，不再把通用配置片段剥离或反向写回。

### 2.3 Gemini 实际落地策略（已补齐）

- Gemini 模板现已接入实际渲染。
- 默认模板拆分为两个逻辑区段：
  - `# .env`
  - `# settings.json`
- 当前支持占位符：
  - `{providerConfig}`：Gemini `.env` 变量
  - `{settingsConfig}`：`settings.json` 中除 `mcpServers` 外的供应商配置字段
  - `{mcpConfig}`：按 Gemini CLI 实际格式渲染后的 `mcpServers`
- 写入流程：
  1. 按模板渲染 `.env` 与 `settings.json`。
  2. `settings.json` 渲染后会再次合并供应商自身配置。
  3. 有启用 MCP 时统一覆盖 `mcpServers`，未启用 MCP 时移除模板中的 `{mcpConfig}` 行并不写空 `mcpServers`。
  4. 渲染结果同时用于：
     - 普通模式写 live
     - 代理接管期间的 Gemini 备份更新
     - 代理接管期间的 Gemini live 安全写入（本地代理地址 + `PROXY_MANAGED`）
     - 恢复时的完整 live 回写

### 2.4 其他应用当前状态

- Claude / OpenCode / OpenClaw / Hermes 也进入统一模板开关链路。
- 默认模板为 `{providerConfig}`，因此默认行为等价于直接写供应商原始配置。
- 若用户自定义模板，系统会先替换 `{providerConfig}`，再交给对应应用的 live 写入适配器。
- 累加模式应用（OpenCode / OpenClaw / Hermes）仍保留多供应商 live 写入模式，模板作用于单个供应商片段，不强行覆盖整个应用文件。

### 2.5 统一机制补充（新增）

- 新增文档：`09-统一配置处理机制.md`
- 重点说明：
  - 六个应用现在共享同一套底层配置处理思路
  - 不同的是最终 live 文件适配器，而不是是否参与配置治理
  - Gemini `.env` 已支持原始文本保真写入，尽量保留注释、分段和顺序

## 3. 当前配置编辑

新增能力：

- 按应用列出一个或多个配置文件（如 Gemini 的 `.env` 与 `settings.json`）。
- 读取文件内容到前端编辑。
- 保存回原文件（含基础路径安全校验）。
- 弹窗直接展开全部文件文本框，不再使用下拉选择。
- 配置模板弹窗提供“加载环境配置”，可从当前应用实际配置文件快速生成模板草稿。

当前版本补充约束：

- `保存` 的语义是“直接写 live 配置文件”，不是自动回写数据库供应商事实源。
- Claude / Codex / Gemini 在代理接管开启时，`保存` 会被禁用，避免绕过接管安全链路直接改写 live。
- 对 Claude / Codex / Gemini 增加显式入口：
  - `导回当前供应商`：把当前配置草稿解析为当前供应商配置，并走现有安全同步链路。
  - `回显到 MCP 管理`：先保存当前 live 草稿，再按应用导入 MCP 到数据库管理页。
- `导回当前供应商` 仅对 `meta.useConfigTemplate = false` 的当前供应商开放。
  - 原因：启用模板时，当前配置看到的是模板渲染结果，不能安全地反向写回供应商 SSOT。

## 4. 启动回显

应用启动时尝试从实际配置解析 MCP 配置并回显至 MCP 管理。

- 支持标准多 `mcpServers` 结构解析。
- 再按应用格式写入对应 `{mcpConfig}`。

## 5. 前后端命令接线

新增 Tauri 命令并在 `invoke_handler` 注册：

- `list_app_config_files`
- `read_app_config_file`
- `write_app_config_file`
- `get_app_config_template`
- `set_app_config_template`

前端通过 `configApi` 对应方法调用，支持“当前配置”弹窗完成文件枚举、读取、编辑、保存闭环。

## 6. MCP 回显增强（与模板关联）

- 启动时 MCP 导入从“仅空表触发”改为“每次启动增量导入”：
  - 文件：`src-tauri/src/lib.rs`
  - 作用：管理页始终能回显外部实际配置变更。
- MCP JSON 解析支持标准 `mcpServers` 包裹结构，并支持多服务器批量解释：
  - 文件：`src/utils/formatters.ts`
  - 表单处理：`src/components/mcp/McpFormModal.tsx`
  - 校验：`src/components/mcp/useMcpValidation.ts`
