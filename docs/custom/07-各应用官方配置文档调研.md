# 各应用官方配置文档调研（2026-04-28）

> 调研日期：2026-04-28  
> 目标：确认各应用官方配置入口与关键字段，为 `{providerConfig}` / `{mcpConfig}` 模板化策略提供依据。

## 1. Claude Code

- 官方设置文档：
  - https://docs.anthropic.com/en/docs/claude-code/settings
- 关键点：
  - 用户级配置文件位于 `~/.claude/settings.json`。
  - MCP 相关能力由 Claude Code 独立机制管理（CC Switch 现有实现也已单独维护 `~/.claude.json`）。
- 对模板策略的启发：
  - `providerConfig` 适合聚焦 `settings.json`。
  - `mcpConfig` 不宜直接混写进 `settings.json`，应继续走独立 MCP 同步链路。

## 2. Codex（OpenAI）

- 官方配置文档：
  - https://github.com/openai/codex/blob/main/docs/config.md
  - https://developers.openai.com/codex/config-reference
- 关键点：
  - 主要配置文件为 `~/.codex/config.toml`。
  - 支持 `model`、`model_provider`、`[model_providers.<id>]`、`[mcp_servers.<id>]` 等结构。
- 对模板策略的启发：
  - `providerConfig` 直接映射到 `config.toml` 的供应商段。
  - `mcpConfig` 可按标准 TOML 子表拼接到 `[mcp_servers.*]`。
  - 必须保持 TOML 有效性与段落稳定，避免结构化 merge 造成乱序。

## 3. Gemini CLI（Google）

- 官方配置文档：
  - https://github.com/google-gemini/gemini-cli/blob/main/docs/reference/configuration.md
- 关键点：
  - 配置覆盖链中包含 `~/.gemini/settings.json`。
  - 实际运行通常还会依赖 `.env` 中的鉴权与环境变量。
- 对模板策略的启发：
  - `providerConfig` 与 `mcpConfig` 应按“`.env` + `settings.json`”双文件视角处理。
  - 不同来源字段应分段覆盖，避免整段覆盖导致配置污染。

## 4. OpenCode

- 官方文档：
  - https://opencode.ai/docs/config
  - https://opencode.ai/docs/providers
  - https://opencode.ai/docs/mcp
- 关键点：
  - 主配置位于 `~/.config/opencode/opencode.json`。
  - 文档明确提供 `provider` 与 `mcp` 配置入口。
- 对模板策略的启发：
  - `providerConfig` 对应 `provider`。
  - `mcpConfig` 对应 `mcp`。
  - 可以天然承载 JSON 模板替换。

## 5. OpenClaw

- 官方文档：
  - https://docs.openclaw.dev/getting-started/configuration
  - https://github.com/openclaw/openclaw/blob/main/docs/gateway/configuration.md
- 关键点：
  - 配置文件路径为 `~/.openclaw/openclaw.json`。
  - 文档包含 `models`、`tools`、`media`（以及网关配置中相关扩展字段）等结构。
- 对模板策略的启发：
  - `providerConfig` 可映射 `models/providers` 相关字段。
  - `mcpConfig` 需结合 OpenClaw 当前版本支持情况按需映射（避免写入无效字段）。

## 6. Hermes Agent

- 官方文档：
  - https://github.com/NousResearch/hermes-agent/blob/main/website/docs/user-guide/configuration.md
- 官方示例（含 `mcp_servers`）：
  - https://raw.githubusercontent.com/NousResearch/hermes-agent/main/website/static/configuration_examples/cli-config.yml
- 关键点：
  - 配置目录与文件通常在 `~/.hermes/config.yaml`。
  - 官方示例包含 `providers` / `model` / `mcp_servers` 等结构。
- 对模板策略的启发：
  - `providerConfig` 与 `mcpConfig` 均可通过 YAML 分段方式映射。
  - 合并时应保留 Hermes 扩展字段，避免丢失用户手工参数。

## 7. 与当前实现的对应关系

- 已稳定落地：
  - Codex：模板持久化 + 实际写入渲染（`build_effective_settings_with_common_config` 统一入口）。
  - 接管代理/故障转移热切换：通过同一有效配置生成路径写入 live 或备份。
- 维持现状（优先稳定）：
  - Claude/Gemini/OpenCode/OpenClaw/Hermes：先保持既有稳定写入与 MCP 同步逻辑。
- 后续扩展建议：
  - 按各应用官方文档逐步把模板渲染从 Codex 扩展到其余应用，并配套“渲染前后 diff”与回滚测试。
