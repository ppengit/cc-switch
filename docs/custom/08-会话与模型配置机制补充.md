# 会话与模型配置机制补充

## 1. 最近会话性能优化

供应商列表顶部“最近会话”改为按当前应用读取最近 10 条，不再扫描全部应用会话后由前端筛选。

后端命令：`list_recent_sessions(appType, limit)`。

效果：

- 减少不必要 I/O。
- 减少前端传输数据量。
- 多应用混用时供应商页打开更稳定。

## 2. 恢复会话交互

如果终端启动失败但恢复命令已经成功复制，只提示“恢复命令已复制”，不再同时提示打开失败，避免语义冲突。

覆盖位置：

- 供应商页最近会话恢复。
- 会话管理页恢复按钮。

## 3. 请求日志会话上下文

请求日志列表展示会话标题和项目路径时，改为批量补充上下文：

1. 按 `app_type + session_id` 聚合当前页会话键。
2. 按应用批量查询标题映射。
3. 回填列表字段。

这样避免分页列表出现 N+1 查询。

## 4. 供应商模型字段的意义

供应商里的模型字段需要保留，但要按运行模式解释。

### 4.1 直连模式

应用直接读取 live 文件，供应商模型通常会成为 CLI 默认模型：

- Claude：`ANTHROPIC_MODEL` 和默认模型映射。
- Codex：`config.toml` 顶层 `model`。
- Gemini：`GEMINI_MODEL` 与 `settings.json.model.name`。

### 4.2 代理接管模式

应用请求进入 cc-switch 后，请求体中的 `model` 优先。供应商模型主要用于：

- 默认模型展示。
- Claude Haiku/Sonnet/Opus 映射。
- 供应商配置恢复。
- 用量统计和价格匹配参考。

### 4.3 故障转移模式

故障转移由代理运行时切换上游供应商：

- live 文件保持本地代理接入配置。
- 上游请求使用被选中供应商的真实配置。
- 请求体模型仍优先，供应商模型用于映射和兜底。

## 5. 请求日志和实时活动的分工

请求日志用于事后审计和详情查看。

实时活动条用于第一时间判断应用是否正在工作、谁在处理请求：

- 不写数据库。
- 请求进入代理即显示。
- 请求结束或异常中断即回落。
- 同时支持多个应用、多供应商并发。
