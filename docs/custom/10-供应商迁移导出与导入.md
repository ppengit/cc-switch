# 10. 供应商迁移导出与导入

## 目标

为本地旧数据库中的供应商体系生成一套可重复执行的迁移包，用于在安装新的定制版本后恢复：

- 供应商列表
- 当前供应商状态
- 故障转移顺序与队列状态
- 供应商相关模板与通用配置
- MCP 服务器配置

同时避免整库导入带来的污染，不覆盖以下数据域：

- 会话
- 请求日志 / 用量统计
- 技能
- 提示词
- 其它非供应商业务表

## 导出范围

导出脚本只处理以下表 / 键：

- `providers`
- `mcp_servers`
- `settings`
  - `common_config_*`
  - `config_template_*`
  - `provider_default_template_*`
  - `current_provider_*`
  - `common_config_legacy_migrated_v1`

## 导入策略

导入不是整库覆盖，而是“按供应商域替换”：

1. 按 bundle 中出现的 `app_type` 清空目标库对应应用的 `providers` 记录。
2. 删除 bundle 内列出的供应商相关 `settings` 键。
3. 若 bundle 包含 MCP 配置，则整体替换 `mcp_servers`。
4. 将导出的供应商域数据重新写入。

这样可以保证：

- 新库中的会话、日志、技能等数据保持不动。
- 新版数据库结构仍然由安装包初始化与迁移逻辑负责。
- 导入动作是可预期、可回滚、可审计的。

## 生成物

导出脚本：`scripts/export_provider_bundle.py`

运行后会在 `release/provider-migration/` 下生成：

- `*.json`
  - 供应商域数据 bundle
- `import_*.py`
  - 面向目标数据库的导入脚本
- `*.README.txt`
  - 本地导入说明

## 安全要求

迁移包可能包含以下敏感信息：

- API Key
- Base URL / 官网地址
- 通用配置片段
- 模板配置
- MCP 配置

因此：

- 迁移包只保留在本地 `release/` 目录
- 不提交到 Git
- 不推送到远程仓库

## 使用流程

1. 运行导出脚本，基于旧库生成迁移包。
2. 安装新的 CC Switch 定制版本。
3. 首次启动一次，让新版本完成数据库初始化。
4. 关闭应用。
5. 运行导入脚本，将供应商域数据写入新库。
6. 导入脚本会自动先备份目标数据库，再执行替换式导入。
