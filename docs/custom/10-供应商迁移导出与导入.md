# 供应商迁移导出与导入

## 目标

为本地数据库中的供应商域生成可重复执行的迁移包，用于安装新版本后恢复供应商数据，同时避免整库导入污染新数据库。

导出内容仅覆盖供应商表：

- 供应商列表。
- 当前供应商状态。
- 故障转移顺序与启用状态。

不导出以下数据：

- settings 表，包括旧通用配置、供应商默认模板、应用接入配置模板。
- MCP 服务器配置。
- 会话。
- 请求日志 / 用量统计。
- 技能。
- 提示词。
- WebDAV。
- 窗口设置。
- 其它非供应商业务表。

## 导出范围

脚本处理以下表：

- `providers`

说明：

- API Key、API 请求地址、模型、协议参数、备注、启用状态、故障转移队列、排序、价格倍率和限额等都在 `providers` 表中。
- 新版应用接入配置模板由安装包默认值和用户之后的编辑决定，不从旧库导入，避免把旧配置机制带回新库。
- 当前供应商通过 `providers.is_current` 恢复；导入脚本会确保每个应用最多一个 current。

## 导入策略

导入不是整库覆盖，而是按供应商应用范围替换：

1. 按 bundle 中出现的 `app_type` 清空目标库对应应用的 `providers` 记录。
2. 写入导出的供应商数据。
3. 按每个应用归一化 `is_current`，防止无 current 或多个 current。

这样可以保证：

- 新库中的会话、日志、技能等数据保持不动。
- 新版数据库结构仍由安装包初始化与迁移逻辑负责。
- 导入动作可预期、可备份、可审计。

## 生成物

导出脚本：`scripts/export_provider_bundle.py`

运行后在 `release/provider-migration/` 下生成：

- `restore_providers_*.py`：自包含目标数据库导入脚本，数据直接嵌入脚本内。
- `*.README.txt`：本地导入说明。

## 安全要求

迁移脚本包含 API Key、Base URL 和完整供应商配置。

因此：

- 迁移包只保留在本地 `release/` 目录。
- 不提交到 Git。
- 不推送到远程仓库。

## 使用流程

1. 运行导出脚本，基于旧库生成还原脚本。
2. 安装新的 CC Switch 定制版本。
3. 首次启动一次，让新版本完成数据库初始化。
4. 关闭应用。
5. 运行还原脚本，将供应商数据写入新库。
6. 导入脚本会自动先备份目标数据库，再执行替换式导入。
