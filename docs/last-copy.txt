## 原因

供应商列表「序号」在故障转移模式下用了两套编号：

```ts
// 旧逻辑
failoverPriority ?? (列表位置 + 1)
```

- **在故障转移队列里**：按队列优先级显示 `1, 2, 3…`
- **不在队列里**：按完整列表位置显示 `1, 2, 3…`

多次启用/禁用、置顶后，禁用项若排在前面，就会出现你说的：

- 禁用 3 个 → `1 2 3`
- 启用 5 个 → 又从 `1` 开始 → `1 2 3 4 5`  
→ 看起来像 **12345，123** 序号乱了、重复。

## 修复

1. **前端序号**：始终按展示队列（`sort_index` 排序后的位置）连续编号 `1…n`，不再混用队列优先级。
2. **新增供应商**：未指定 `sort_index` 时，后端自动追加到当前应用末尾。
3. **编辑供应商**：提交时若未带 `sort_index`，保留数据库原值，避免被清空。
4. **测试**：覆盖「禁用在前 + 启用在后」场景，断言序号为连续 `1…8`。

- `src/components/providers/ProviderList.tsx`

### A. 统一配置处理机制（核心，上次说明不足）

设计定稿见 `docs/custom/03`、`09`、`11`：

**三个事实源（不可混）**

**两条写入链路**

| 模式 | live 写什么 | 真实 Key/地址 |
|------|-------------|---------------|
| 普通直连 | 当前供应商真实配置 + 启用 MCP | 写进应用 live |
| 代理接管 | 接入模板渲染结果（本地代理 + `PROXY_MANAGED`） | 只在 DB / live backup / 代理转发时用 |

**本地特有/强化能力**

- 后端默认模板收敛：`src-tauri/src/app_config_templates.rs`（**上游没有该文件**）
- 多文件模板：`config_template_{app}` → `AppConfigTemplateFile[]`（Claude / Codex auth+config / Gemini env+settings…）
- 占位符：`{proxyBaseUrl}` / `{proxyCodexBaseUrl}` / `{proxyToken}` / `{mcpConfig}`
- **当前配置**弹窗：直接展开全部 live 文件编辑，非单文件下拉  
- 接管中禁止直写 live 绕过（防泄真实 Key）  
- 模板保存 / MCP 变更 / 热切换：先更新 **live backup**，再重建接管 live  
- 故障转移：**不靠改写 CLI live 切供应商**，靠代理运行时选上游  
- 配置命令面明显更重：`get/read/write_app_config_file`、`get/set_app_config_template` 等（`config.rs` 体量差 3～4 倍）

- MCP 删了又复活 / 整文件被清空
